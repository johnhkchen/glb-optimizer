## Context

The accepted-tag is the most valuable training signal in the entire epic. When a user marks an asset as "this is good, ship it," we snapshot the current settings and tag them as the canonical config for that asset. Later ML can train on `(asset features → accepted settings)` pairs.

This ticket also delivers the export script — a way to dump all accumulated tuning data into a clean JSONL bundle suitable for ML training or external analysis.

## Acceptance Criteria

- New `accepted_settings.json` per asset stored at `~/.glb-optimizer/accepted/{asset_id}.json`:
  - Schema: `{asset_id, settings: AssetSettings, accepted_at, comment, thumbnail_path}`
  - Immutable snapshot — re-accepting writes a new version (or replaces; user choice — pick the simpler one for v1, replacement is fine)
- HTTP endpoint `POST /api/accept/:id` snapshots the current settings + writes the file + emits an `accept` analytics event
- "Mark as Accepted" button in the tuning panel (or near the file's status badge)
  - Optional comment input
  - Confirmation of save
- When an asset is accepted, its file list entry shows a small ✓ marker
- Linked thumbnail: `accept` events grab a 256px JPEG render of the current preview canvas and save to `~/.glb-optimizer/accepted/thumbs/{asset_id}.jpg`
- Export script `scripts/export_tuning_data.py` (or Go equivalent) that produces:
  - A single JSONL file aggregating all sessions, all events, all profiles, all accepted snapshots
  - Includes file paths to linked thumbnails (relative or absolute, document the choice)
  - Output schema matches `analytics-schema.md`'s "export format" section (add this section if missing from T-003-01)
- Manual verification: accept an asset, run the export script, confirm the output JSONL contains the accept event AND references the thumbnail

## Out of Scope

- ML training pipelines (just produce the data, don't consume it)
- Versioned accept history (single canonical accepted snapshot per asset)
- Cloud upload of exports
- Aggregation dashboards

## First-Pass Scope

The export script can be a single file with no dependencies beyond stdlib + maybe a JSON helper. Output format matters more than the script's polish.
