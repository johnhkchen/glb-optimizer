## Context

When the classifier from T-004-02 returns a confidence below threshold (~0.7), we want to render the asset under 2-3 candidate strategies side-by-side and let the user pick. The pick is the most valuable training signal in this story — every comparison the user resolves becomes a labeled training example linking `(asset features → human-preferred strategy)`.

## Acceptance Criteria

- When `POST /api/classify/:id` returns confidence < 0.7, the frontend automatically opens a comparison view
- Comparison view:
  - Renders the asset under 2-3 candidate strategies (e.g., `round-bush`, `directional`, `tall-narrow`)
  - Each candidate is a thumbnail (~256px) showing the asset processed with that strategy
  - User picks one
- Picking a strategy:
  - Sets the asset's `shape_category` to the chosen value
  - Sets `shape_confidence` to 1.0 (human-confirmed)
  - Stores a `classification_override` analytics event with: original category, confidence, all candidate categories shown, chosen category, asset features
- Comparison view also accessible from the tuning panel as a "Reclassify..." action — user can override even confident classifications
- The thumbnails are generated via the existing offscreen render pipeline; expect this to take a few seconds
- Manual verification: pick an ambiguous asset (use a tall-but-bushy plant), confirm comparison UI opens, pick a strategy, confirm the classification override is stored and the analytics event is logged

## Out of Scope

- Trellis end-to-end validation (T-004-05)
- Real-time preview as the user hovers candidates
- Confidence threshold tuning (start at 0.7, adjust later based on override rates)
- Branching topology candidates

## First-Pass Scope

The comparison UI is the main deliverable. It can be a simple modal with three thumbnails and three buttons. No fancy layout, no carousel, no zoom. The analytics events matter more than the UX polish — that's where the training data lives.
