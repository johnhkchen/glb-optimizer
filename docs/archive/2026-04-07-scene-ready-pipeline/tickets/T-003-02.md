## Context

T-003-01 built the storage layer. This ticket wraps the existing settings system from S-002 with automatic event capture. Every interaction in the tuning UI fires a `setting_changed` event without each control needing manual instrumentation.

Sessions wrap the user's interaction with one asset: `session_start` when a file is selected, `session_end` when another file is loaded or the tab closes. The session abstraction is what lets us correlate "user changed dome height, then regenerated, then changed it again" into a meaningful sequence.

## Acceptance Criteria

- Frontend session lifecycle:
  - `selectFile(id)` calls `startAnalyticsSession(id)` (which calls the backend to create a session ID)
  - When selecting a different file or closing the tab, `endAnalyticsSession(outcome)` fires with `outcome` = `switched` | `closed`
  - Session ID stored in a module-level variable
- Auto-instrumentation of the tuning panel from T-002-03:
  - Wrap the debounced settings PUT so it ALSO fires a `setting_changed` event
  - Payload: `{key, old_value, new_value}`
  - Include the timestamp delta from the previous setting change in the same session (helps detect "fast revert" later)
- Auto-instrumentation of regenerate actions:
  - Every call to `generateBillboard`, `generateVolumetric`, `generateProductionAsset`, etc. fires a `regenerate` event
  - Payload: `{trigger: "billboard"|"volumetric"|"production", success: bool}`
- The `selectFile()` function loads any previous session for the asset (or starts a new one if none exists)
- Manual verification: load asset → change three sliders → click Production Asset → load another asset. Confirm the JSONL file contains: session_start, 3× setting_changed, regenerate, session_end.

## Out of Scope
