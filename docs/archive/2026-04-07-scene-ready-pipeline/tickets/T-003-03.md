## Context

Profiles are named bundles of settings the user can save, load across assets, and leave comments on. They're distinct from per-asset settings (mutable, scoped) and accepted-tags (immutable snapshots). Profiles are how a user captures "this lighting + dome config worked great for round bushes" and reuses it.

The user's framing: "profiles we can reason about, and let user leave comments alongside."

## Acceptance Criteria

- New Go module `profiles.go` with:
  - `Profile` struct: `{name, comment, settings: AssetSettings, created_at, source_asset_id}`
  - `LoadProfile(name)` from `~/.glb-optimizer/profiles/{name}.json`
  - `SaveProfile(profile)` writes atomically
  - `ListProfiles()` returns all profile metadata (name + comment, no full settings)
  - `DeleteProfile(name)`
- HTTP endpoints:
  - `GET /api/profiles` returns the list
  - `GET /api/profiles/:name` returns full profile
  - `POST /api/profiles` accepts `{name, comment, settings}`, saves
  - `DELETE /api/profiles/:name`
- New section in the tuning panel (or modal): "Profiles"
  - Dropdown listing existing profiles
  - "Save current as profile..." action with name + comment input
  - "Apply profile" button — overwrites current asset settings with profile settings
  - "Delete profile" action
- Applying a profile fires a `profile_applied` analytics event with `{profile_name}`
- Saving a profile fires a `profile_saved` event
- Profile names: kebab-case, validated server-side, max 64 chars

## Out of Scope

- Profile sharing / export (use the file directly for now)
- Profile templates / built-in starter profiles
- Profile versioning (overwrite is fine for v1)
- Search / filter / tag profiles

## First-Pass Scope

Functional CRUD + a simple UI section. No fancy modal, no preview thumbnails, no rich-text comments. A textarea for the comment is enough.
