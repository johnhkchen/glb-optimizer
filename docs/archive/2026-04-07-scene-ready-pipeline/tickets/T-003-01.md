## Context

First brick of the analytics foundation. Define the versioned event schema and the local JSONL storage layer. No instrumentation hooks yet (T-003-02), no profiles (T-003-03), no UI surface. Just: events get appended to disk in a stable format.

The schema decision is permanent — once events start landing on disk under v1, every future change has to migrate. Get the field names right and version the envelope.

## Acceptance Criteria

- `docs/knowledge/analytics-schema.md` documents the event format:
  - Envelope: `{schema_version, event_type, timestamp, session_id, asset_id, payload}`
  - Initial event types: `session_start`, `session_end`, `setting_changed`, `regenerate`, `accept`, `discard`
  - Each event type's payload schema with field types
- New Go module `analytics.go` with:
  - `Event` struct mirroring the envelope
  - `AppendEvent(sessionID, event)` writes to `~/.glb-optimizer/tuning/{session_id}.jsonl` (one event per line)
  - `StartSession(assetID)` → returns new session ID (UUID)
  - Atomic append (open append + flush per write) so concurrent writes don't corrupt
- HTTP endpoint `POST /api/analytics/event` accepts an event JSON, validates schema_version, appends
- Tuning directory created at startup if missing
- Frontend helper `analytics.js` (or section in `app.js`) with `logEvent(type, payload)` that POSTs to the endpoint
- Manual verification: trigger a setting change in T-002-03's UI, confirm a JSONL line lands on disk

## Out of Scope

- Auto-instrumentation of settings changes (T-003-02)
- Session lifecycle hooks (T-003-02)
- Profiles, accepted-tag, export script (T-003-03, T-003-04)
- Real-time event viewer
- Analytics dashboards

## First-Pass Scope

Integration over polish. The schema must be coherent and stable, but the capture API can be minimal — no batching, no retries, no compression. Direct POST per event is fine for v1.
