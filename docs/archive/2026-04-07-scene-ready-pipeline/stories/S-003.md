## Analytics Foundation

### Context

Every tuning interaction is potential training data. A landscape designer who slides "dome height" from 0.5 to 0.7, regenerates, decides it's better, and saves — that's a labeled example. A user who tries three different lighting presets and picks the second one — that's a 3-way preference. Multiplied across thousands of users and tens of thousands of assets, this is the corpus that turns the pipeline from "tunable by hand" into "auto-tuned by ML."

This story builds the foundation. It must ship before the tuning controls in S-005, S-007 do — otherwise we lose the data from those first cohort sessions.

Storage is local-only for now (`~/.glb-optimizer/tuning/`). The format is spec'd to be uploadable to a central store later without schema changes.

### Acceptance Criteria

- Versioned event schema documented in `docs/knowledge/analytics-schema.md`
- Local JSONL storage at `~/.glb-optimizer/tuning/{session_id}.jsonl`
- **Session abstraction**: a tuning session starts when the user opens an asset and ends when they save/discard/leave. Each session has a UUID and links to the asset
- Every settings change emits a `setting_changed` event with: timestamp, session, asset_id, key, old_value, new_value
- Every regenerate emits a `regenerate` event with: trigger (which control), output thumbnail path, output GLB path
- Every accept/discard emits a `session_end` event with: outcome, final settings, optional comment
- **Tagged final settings**: when a user marks an asset as accepted, the current settings are stored as the canonical "accepted profile" for that asset. This is the most valuable training signal — record it explicitly
- **Profiles** are first-class:
  - Save current settings as a named profile (with optional comment)
  - List existing profiles
  - Apply a profile to the current asset
  - Profiles stored in `~/.glb-optimizer/profiles/{profile_name}.json`
  - Profile schema includes: name, comment, base settings, created_at, source asset (if applicable)
- **Satisfaction signals** captured implicitly:
  - Time between regenerate and next change (fast = dissatisfied, slow = inspecting)
  - Did the user revert a setting within N seconds of changing it
  - Did the session end with save vs discard
- Linked thumbnails: each `regenerate` event references a small (256px) preview rendered to `~/.glb-optimizer/tuning/thumbs/{event_id}.jpg`
- Export script: `scripts/export_tuning_data.py` produces a clean JSONL ready for ML training (events + linked artifacts + asset features placeholder for S-004 to fill in)

### Out of Scope

- Centralized upload / cloud storage
- Dashboards beyond a basic event log viewer
- Multi-user analytics (no auth, no user ID — single-user tool for now)
- Privacy controls (single-user, all local)
- Real-time aggregation

### Notes

- The schema is the most important deliverable. Get it reviewed before implementing the capture layer. Once events are recorded under v1, we have to support migration forever.
- Profiles vs settings vs accepted-tag: settings are per-asset and mutable; profiles are named, reusable, and shareable; accepted-tags are immutable snapshots that mark "this is the final canonical config for this asset." All three concepts coexist.
- Resist building a fancy UI for analytics in this story. A bare event log + profile manager is enough. Visualization is a follow-up epic.
