## Tilted Billboard Transition Band

### Context

The current production asset has two impostor types that the runtime crossfades between as the camera tilts:

- **Camera-facing billboards** — rendered straight-on (camera elevation 0°), visible at low camera tilt
- **Overhead dome slices** — rendered top-down (camera elevation 90°), visible at high camera tilt
- **Crossfade band** — `lookDownAmount` 0.55-0.85 (~33-58° tilt)

At the crossfade band, neither representation is visually correct. Billboards look like they're tipping over because they're being rendered head-on but viewed from above. Dome slices look paper-thin because they're being viewed nearly edge-on. The result is a noticeable visual artifact band — exactly the part of the camera arc a landscape designer is most likely to use when previewing a planted bed.

### Solution

Introduce a third impostor type: **tilted billboard**, rendered with the camera elevated ~30-35° above horizontal, still in N azimuth variants and still camera-facing in yaw at runtime. Then split the existing 2-state crossfade into three stages:

```
lookDownAmount: 0.0 ─── 0.3 ─── 0.55 ─── 0.75 ─── 1.0

  billboards:   ████████▓▓▒▒░░
  tilted:                ░░▒▒▓▓████▓▓▒▒░░
  dome slices:                       ░░▒▒▓▓████
```

- **0.00-0.30**: billboards solo (camera roughly horizontal, classic side view)
- **0.30-0.55**: crossfade billboards → tilted (band 1, centered ~24° tilt)
- **0.55-0.75**: tilted solo (camera ~38-49° tilt — the "looking-around" angle)
- **0.75-1.00**: crossfade tilted → dome slices (band 2, centered ~49° tilt)

### Reuse

`renderBillboardAngle` already takes a camera angle. Generalizing it to also accept an **elevation** parameter is mechanical — the existing function positions the camera around the model in the X-Z plane at the model's center Y; the new variant adds a Y component. Everything downstream (storage, upload endpoint, instance creation, LOD button registration, crossfade math) mirrors the existing billboard pipeline 1:1.

### Memory Budget

Each tilted-billboard variant is one quad per azimuth angle. At 6 angles × 512px JPEG texture (~50 KB each), that's ~300 KB extra per asset. The current production asset bundle is typically 500 KB-2 MB; adding 300 KB is a ~15-30% increase. Acceptable.

### Out of Scope

- Multiple tilt elevations (just one tilted variant for now — 30-35° feels right; tunable later)
- Adaptive tilt elevation per shape category (round-bush wants different tilt than tall-narrow — follow-up if it matters)
- Per-instance tilt jitter
- Tilted variant for the dome slices (would be a "tilted dome" — different problem, separate story if needed)
- Replacing the existing horizontal billboards (the tilted variant is additive — both still ship)

### Notes

- Single-tilt-elevation is the v1 simplification. The bake function should accept the elevation as a parameter so a follow-up can add multiple elevations cheaply.
- The crossfade bands are best-guess starting points. They should land as tunable settings (`tilted_fade_low`, `tilted_fade_high`) so the designer can dial them in by eye against a real bed scene.
- The tilted billboards should fire all the same analytics events as the horizontal billboards (`regenerate` with a `trigger: "billboard_tilted"` payload) so we can correlate user satisfaction with the new impostor type vs the old.
