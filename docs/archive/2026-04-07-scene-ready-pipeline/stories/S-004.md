## Geometry-Aware Asset Classification

### Context

The pipeline currently treats every asset as a round bush with rotational symmetry. The trellis incident exposed this: trellised peas got per-instance Y rotation and produced an incoherent mess. Trellises, fences, espaliered fruit trees, ornamental grasses, and any directional asset all break the same way.

The fix is to **infer the asset's shape from its geometry** and route to a strategy that suits it. We're not asking users to label assets — we already have the high-poly model on disk, and "we start with models and arrive at budget assets" means the model itself is the source of truth for shape category.

The approach is PCA-style: compute principal axes of the vertex cloud, look at the axis-magnitude ratios, classify into a small starter taxonomy. When the classifier is uncertain, render multiple candidate strategies side-by-side and let the user pick — every comparison becomes a labeled training example for analytics.

### Acceptance Criteria

- New analysis module (Python script or Go module) that consumes a GLB and outputs:
  - Principal axis vectors (3) and magnitudes (3)
  - Axis-magnitude ratios
  - Aspect ratio (height / max-horizontal)
  - Vertex density distribution along each principal axis
  - Classification: one of `round-bush`, `directional`, `tall-narrow`, `planar`, `hard-surface`, `unknown`
  - Confidence score in [0, 1]
- Per-category strategy router that maps a category to:
  - Slice axis (which principal axis to slice along)
  - Slice count and distribution
  - Instance orientation rule (random Y / fixed / aligned-to-row / random within ±N°)
  - Default budget allocation
- **Multi-strategy comparison UI** for ambiguous cases (confidence < threshold):
  - Render the asset with 2-3 candidate strategies
  - Show side-by-side thumbnails
  - User picks one
  - Pick is recorded as a `classification_override` analytics event
- Trellis test case: trellised pea model is classified as `directional`, instances stay aligned (no per-instance Y rotation), a row of 5 trellises looks like a row
- Round bush test case: rose bush is classified as `round-bush`, instances get random Y rotation, behavior unchanged from current
- Hard-surface test case: wooden raised bed is classified as `hard-surface`, gets the parametric pipeline from S-001
- Classification result stored on the asset's settings (from S-002) so it's available downstream
- Every classification emits a `classification` analytics event with all features and chosen category

### Out of Scope

- Branching topology (Y-shaped plants, candelabra shapes) — stretch goal, follow-up epic
- Hardscape sub-categories (water features, vapors, patio surfaces) beyond the basic `hard-surface` bucket
- Time-series shape changes (seasonal foliage, growth stages)
- Per-vertex semantic labels

### Notes

- **Spike subtask up front**: before fully implementing, run PCA on the existing reference assets (`rose_julia_child.glb`, `wood_raised_bed.glb`, and a trellis if available) and verify the axis-magnitude ratios meaningfully distinguish them. If naive PCA fails, the fallback is bounding-box ratios + a density histogram. Document the spike result in `docs/active/work/S-004/research.md`.
- The classifier will be wrong sometimes. The multi-strategy comparison UI is the safety net AND the data collection mechanism. Don't try to make the classifier perfect — make the override flow smooth.
- Confidence threshold for triggering the comparison UI starts conservative (~0.7) and gets tuned based on how often users override.
