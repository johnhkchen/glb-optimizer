## Settings Object + Tuning UI Skeleton

### Context

Today, every parameter that affects the bake (slice count, dome height, exposure, ambient intensity, key light intensity, env map intensity, alpha test thresholds, etc.) is a hardcoded constant in `static/app.js`. To tune the pipeline, the user has to ask Claude to edit constants in source. This is the bottleneck that blocks every other story in this epic.

This story moves all those constants into a per-asset **settings object**, persists it alongside the GLB, and exposes a basic UI skeleton for editing them. No fancy presets, no analytics yet — just the bones. Default values exactly reproduce current behavior so existing assets render identically.

### Acceptance Criteria

- Settings JSON schema defined and documented (one canonical version per asset)
- Per-asset settings stored in `~/.glb-optimizer/settings/{asset_id}.json`
- Backend endpoint to read/write per-asset settings
- Settings drive both the bake AND the live preview consistently — no separate "preview" vs "bake" parameters
- New right-panel UI section "Tuning" with sliders/dropdowns for at minimum:
  - Slice count
  - Dome height factor
  - Exposure (tone mapping)
  - Ambient intensity
  - Key light intensity
  - Hemisphere intensity
  - Env map intensity
  - Alpha test threshold
- All existing behaviors reproducible via default settings (regression test: re-bake an existing asset with defaults, compare to pre-change output)
- Settings persist across server restarts
- Selecting a file loads its saved settings into the UI
- "Reset to defaults" action

### Out of Scope

- Named profiles (S-003)
- Analytics event capture (S-003)
- New tuning controls beyond the existing constants (S-005)
- Auto-classification (S-004)
- Lighting presets (S-007)

### Notes

- This is the lowest-glamour story in the epic and the highest leverage. Get the schema right.
- Schema should be versioned (`schema_version: 1`) so future changes can migrate cleanly.
- Resist adding new sliders here — that's S-005's job. This story is purely an inversion: hardcoded → settings object.
