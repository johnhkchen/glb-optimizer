# Research — T-003-04: accepted-tag-and-export

## Ticket scope (recap)

Two distinct deliverables in one ticket:

1. **Accept-as-canonical** — a per-asset "this is shippable" snapshot of
   the current `AssetSettings`, plus a 256px JPEG thumbnail of the
   preview canvas at accept time, persisted at
   `~/.glb-optimizer/accepted/{asset_id}.json` and
   `~/.glb-optimizer/accepted/thumbs/{asset_id}.jpg`. Wired through:
   - a backend endpoint `POST /api/accept/:id`
   - an `accept` analytics event (already in `validEventTypes`)
   - a "Mark as Accepted" button in the tuning panel
   - a ✓ marker in the file list when the asset is accepted

2. **Export script** — `scripts/export_tuning_data.py` (Python by
   ticket preference) that aggregates *every* tuning artifact on disk
   into a single JSONL bundle suitable for ML training.

The ticket calls out that the export-format section may need to be
**added** to `analytics-schema.md` (T-003-01 left it out).

## Current on-disk layout

The Go server (`main.go:56-67`) creates and owns the following
subdirectories under `~/.glb-optimizer/` (or `--dir`):

```
~/.glb-optimizer/
    originals/{id}.glb              # uploaded source GLBs
    outputs/{id}.glb                # gltfpack outputs + _billboard, _volumetric, _vlod0..3
    settings/{id}.json              # per-asset AssetSettings (T-002-01)
    tuning/{session_id}.jsonl       # per-session analytics events (T-003-01)
    profiles/{name}.json            # named Profile snapshots (T-003-03)
```

T-003-04 introduces a sixth top-level subdirectory:

```
    accepted/{id}.json              # accepted snapshot (this ticket)
    accepted/thumbs/{id}.jpg        # 256px JPEG capture
```

`{id}` is the same 32-char hex generated by `generateID()` in
`handlers.go:21-25` and is the file id used everywhere else.

## Existing analytics + accept-event surface

`analytics.go` already accepts `accept` events end-to-end:

- `validEventTypes` (`analytics.go:25-34`) includes `"accept": true`.
- `analytics-schema.md` documents the `accept` event payload as
  `{"settings": {...AssetSettings...}}`. The thumbnail field is
  already reserved on `regenerate` (`thumbnail_path`) but not on
  `accept`.
- `app.js` does **not** currently emit any `accept` event. There is
  no UI for it. The string `'accept'` only appears in `app.js` as a
  hypothetical `outcome` for `session_end` (in the doc comment) and
  in the pipeline name "accepted-tag" — the actual event-type emitter
  has to be added by this ticket.

The frontend's `logEvent(type, payload, assetId)` helper
(`app.js:218-238`) is the only thing T-003-04 needs to call to land
an event in the right session JSONL — the helper handles the
envelope, the session id, and the wire format.

## Settings + Profiles already provide most of the wiring

The accepted-snapshot file is structurally a near-clone of `Profile`
in `profiles.go`:

- `Profile` already wraps `*AssetSettings` plus metadata
  (`name`, `comment`, `created_at`, `source_asset_id`).
- `SaveProfile` writes via `writeAtomic` (`settings.go:172-194`).
- `LoadProfile`, `ListProfiles`, `DeleteProfile` exist.

The accepted snapshot differs from a profile in three ways:
1. The filename is the **asset id**, not a kebab-case name.
2. There is one canonical accepted snapshot per asset; re-accepting
   *replaces* (ticket: "replacement is fine" for v1).
3. It carries a `comment` and a `thumbnail_path` field, but no
   `name` field.

The `FileRecord` struct (`models.go:45-62`) already exposes a
boolean `HasSavedSettings`. Adding `IsAccepted bool` follows the
same pattern: derived at scan time from the existence of the
accepted snapshot file, surfaced via `/api/files`.

`scanExistingFiles` in `main.go:161-194` is the single startup hook
that populates these flags. It already takes `settingsDir` to set
`HasSavedSettings`; it will need an `acceptedDir` parameter as well.

## Frontend tuning panel surface

The right panel (`static/index.html:94-330`) holds the
`#tuningSection` (lines 219-304) followed by `#profilesSection`
(lines 306-329). The Accept button most naturally sits as a third
section *after* profiles, or as a single new row inside
`#tuningSection`. Either is one HTML insertion + one event listener
+ one Go round-trip.

`selectFile` (`app.js:2521-2551`) is the central place where
per-asset state is rebuilt on click. The accept button needs to
become enabled/disabled here based on whether a file is currently
selected, and re-render after a successful accept (so the ✓ marker
flips on without a full reload).

The file list rendering in `renderFileList` (`app.js:1761-1827`)
formats `metaHTML` from the file record. Adding a `is_accepted`
indicator is a one-line append next to `status-badge`.

## Preview canvas → thumbnail capture

The preview canvas is `#preview-canvas` (`static/index.html:87`),
managed by Three.js (`renderer = new THREE.WebGLRenderer({ canvas:
previewCanvas, ... })` at `app.js:1835`). Capturing a 256px JPEG
from it has two viable strategies:

A. **Client-side capture, base64 in POST body.** Call
   `previewCanvas.toDataURL('image/jpeg', 0.85)` (or
   `toBlob()`), downsample to 256 longest-edge in a temp
   off-screen canvas, and POST it as a multipart field or as a
   data URL alongside the accept request. The browser already
   has a fully-rendered scene at that moment; this is the simplest
   path. ⚠️ Three.js `WebGLRenderer` may need
   `preserveDrawingBuffer: true` for `toDataURL` to read back
   pixels reliably; the current renderer is constructed without
   that flag (`app.js:1835`). A workaround is to call
   `renderer.render(scene, camera)` immediately before reading
   the buffer, while the GL context is still in the same frame
   tick.

B. **Server-side capture.** The Go server has no headless
   renderer; it would have to invoke Blender or some other
   external tool. Way out of scope for "save a thumbnail."

→ Strategy A is the obvious pick. The thumbnail capture is a
client concern; the server just receives bytes and writes them.

## Export script — language and packaging

The `scripts/` directory already contains
`bake_textures.py`, `parametric_reconstruct.py`, `remesh_lod.py`
— so Python is the established convention. The ticket also says
"Go equivalent" is acceptable, but Python is the path of least
surprise.

The script must read from `~/.glb-optimizer/` (or a
`--dir` arg matching the server's flag) and produce a JSONL
file. Inputs:

- `tuning/*.jsonl` — concat all events
- `settings/*.json` — current per-asset settings
- `profiles/*.json` — named profiles
- `accepted/*.json` — accepted snapshots
- `accepted/thumbs/*.jpg` — referenced (path only, not embedded)

The output schema is whatever this ticket defines as the
"export format" section that's missing from `analytics-schema.md`.

The "thumbnail path" representation (relative or absolute) is a
design decision the ticket explicitly defers to us, with the
single requirement that the choice be documented.

## Constraints and assumptions

- **Single-user, local-only.** No auth, no rate limiting, no
  multi-tenant concerns. Same as every other endpoint in the
  project.
- **No third-party Go dependencies.** `go.mod` is stdlib-only;
  the precedent (T-003-01..03) is to keep it that way.
- **Python script: stdlib only.** The ticket explicitly says
  "no dependencies beyond stdlib + maybe a JSON helper."
- **Schema-version stability.** The accepted snapshot schema
  must declare its own `schema_version`, parallel to
  `AssetSettings` and `Profile`. Future bumps follow the same
  policy as the rest of S-003.
- **Atomicity on write.** Use `writeAtomic` (already in
  `settings.go`) for the JSON snapshot. The thumbnail JPEG is
  written via the standard "create-temp + rename" idiom or
  `os.WriteFile` (single-pass writes are fine for binary blobs
  in v1; if torn writes ever bite, switch to atomic).
- **Manual verification only.** The ticket's "Manual verification"
  section is the de facto e2e test: accept an asset, run the
  export script, confirm the JSONL contains the accept event and
  the thumbnail reference.

## Open questions for the design phase

1. Should the accepted snapshot file's schema be a thin wrapper
   that *embeds* `AssetSettings` (like `Profile`) or a flat
   structure with the settings spread inline? Profile does
   the wrapper approach; consistency suggests the same here.
2. Where in the right panel should the "Mark as Accepted" button
   live — top of the tuning section (most prominent), inside
   profiles (least intrusive), or as its own third
   `.settings-section` (clearest visual separation)? Each is one
   line of HTML.
3. Thumbnail upload: multipart form vs. base64 inside the JSON
   body. Multipart is more bytes-on-the-wire-efficient and
   matches `handleUpload` already in the codebase. Base64 is
   simpler and the payload is small (256px JPEG ~10–20 KB).
4. Thumbnail path in export: relative to `~/.glb-optimizer` (so
   `accepted/thumbs/{id}.jpg`) or absolute (so users can `cp` the
   JSONL elsewhere and still resolve)? The relative form is
   portable; absolute is robust against move-after-export.
5. The export script's filename format: what does the resulting
   bundle JSONL look like? One record per asset (joined view) vs.
   one record per event (raw stream)? The ML training framing in
   the ticket leans toward "one record per accepted asset" as
   the primary product, with raw events available as a side
   stream.

These are the explicit decision points for `design.md`.
